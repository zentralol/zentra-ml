# Zentra Crowd Prediction — ML Backend

Predicts pedestrian crowd levels at any location in Manhattan using a machine learning ensemble trained on 18 years of NYC open data. Exposes predictions through a REST API built with FastAPI.

## Project Overview

The system has two parts:

**1. Notebooks** — run once to process raw data and train models  
**2. API** — reads the trained model files and serves predictions on demand

```
Raw NYC data  →  [Data Processing Notebook]  →  Processed CSVs + Master dataset
                                                        ↓
                       [Crowd Prediction Notebook]  →  Trained model files
                                                        ↓
                              [FastAPI (api/)]  →  REST API endpoints
```

## How the Notebooks Work

### Notebook 1 — `crowd_prediction_data_processing.ipynb`

Reads everything in `data/raw/` and produces clean, merged files in `data/processed/` and `data/master/`.

**What it does, section by section:**

| Section | What it does |
|---------|-------------|
| 1. H3 Grid | Creates a 524-cell hexagonal grid over Manhattan at H3 resolution 9 (~150m cells) |
| 2. Pedestrian Counts | Cleans the NYC DOT bi-annual survey data, maps each survey location to an H3 cell |
| 3. TLC Taxi Data | Aggregates yellow cab + rideshare (HVFHV) trip pickups/dropoffs per H3 cell per period |
| 4. MTA Ridership | Maps subway station ridership to the nearest H3 cell |
| 5. Citi Bike | Aggregates Citi Bike trip starts/ends per H3 cell |
| 6. Weather | Pulls historical weather for each survey date; builds a seasonal average reference |
| 7. POI (Points of Interest) | Counts restaurants, subway stops, offices, retail, tourism, nightlife per H3 cell using OpenStreetMap |
| 8. Events | Processes NYC event data into a daily intensity score |
| 9. Calendar Features | Adds day of week, weekend flag, holiday flag, school break flag, cyclical month/period encodings |
| 10. Master Join | Joins all sources on (h3_cell, period, survey_date) into one master training table |
| 11. Live Grid Snapshot | Scores all 524 Manhattan cells with current mobility + POI signals (used by the API for untrained locations) |
| 12. Lag Features | Adds corridor-level temporal features: ped_lag1, ped_lag2, rolling mean/std, YoY ratio, expanding mean |

---

### Notebook 2 — `crowd_prediction.ipynb`

Reads the master dataset from notebook 1, trains three ML models using Optuna hyperparameter tuning, blends them into an ensemble, and exports all model files.

**What it does, section by section:**

| Section | What it does |
|---------|-------------|
| 1. Load Data | Reads `zentra_training_master.csv`, applies log1p transform to pedestrian counts |
| 2. Feature Selection | Defines the 81 features used for training (FEATURE_COLS) |
| 3. Train/Val Split | Time-based split — last year of data held out as validation |
| 4. Baseline Models | Quick comparison of 5 models (LGB, XGB, RF, GB, Ridge) to pick the best combination |
| 5. LightGBM Tuning | Optuna with 100 trials, TimeSeriesSplit(5) cross-validation |
| 6. HistGradientBoosting Tuning | Optuna with 80 trials |
| 7. RandomForest Tuning | Optuna with 40 trials (uses median imputer since RF can't handle NaN) |
| 8. OOF Predictions | Out-of-fold predictions for all 3 models (used for ensemble weight optimisation) |
| 9. Ensemble Weights | scipy.optimize.minimize (Nelder-Mead) finds the optimal blend weights for each task |
| 10. Evaluation | Computes MAE, RMSE, R², MAPE on validation set for each model and the ensemble |
| 11. Plots | Pred vs actual, residual analysis, SHAP feature importance, model comparison |
| 12. Batch Scoring | Scores all 524 Manhattan cells using the live grid template |
| 13. Export | Saves all model files to `models/` |

### Generated by Notebook 2 (Model Training)

Saved to `models/`:

| File | Description | Used by |
|------|-------------|---------|
| `lgbm_tuned.txt` | Trained LightGBM model (text format) | **API** (`inference.py`) |
| `gb_tuned.pkl` | Trained HistGradientBoosting model | **API** (`inference.py`) |
| `rf_tuned.pkl` | Trained RandomForest model | **API** (`inference.py`) |
| `rf_imputer.pkl` | Median imputer fitted on training data (required before RF predictions) | **API** (`inference.py`) |
| `ensemble_weights.json` | Optimised blend weights for crowd task and future task, plus HIST_P95 value | **API** (`inference.py`) |
| `inference_bundle.joblib` | Feature column list (FEATURE_COLS), training medians, missing-flag columns, dummy groups | **API** (`inference.py`) |

**`ensemble_weights.json` structure:**
```json
{
  "crowd":  { "lgb": 0.31, "gb": 0.42, "rf": 0.27 },
  "future": { "lgb": 0.40, "gb": 0.60 },
  "hist_p95_log": 8.74
}
```

- `crowd` weights are used by `POST /predict/crowd` — 3-model blend
- `future` weights are used by `POST /predict/future` — 2-model blend (no RF)
- `hist_p95_log` is the 95th percentile of log(pedestrians) in training — used to normalise predictions to a 0–100 score

## How to Run the API

### Prerequisites — Set up a virtual environment

Run the project inside a virtual environment so dependencies stay isolated. Pick **one** of the options below, then install from `requirements.txt`.

#### Option A — uv (recommended)

```bash
# from the zentra-ml/ project root
uv venv                        # creates .venv with the project's Python
source .venv/bin/activate      # Windows: .venv\Scripts\activate
uv pip install -r requirements.txt
```

#### Option B — conda

```bash
conda create -n zentra python=3.11 -y
conda activate zentra
pip install -r requirements.txt
```

### Step 1 — Run the notebooks (optional, first time only)

> **Optional.** The repo already ships the static files these notebooks produce — the
> processed CSVs in `data/processed/` and `data/master/`, plus the trained models in
> `models/`. The API runs directly off those, so you do **not** need to retrain on every
> setup. Only run the notebooks if you want to regenerate the data or retrain the models
> from scratch (e.g. after updating the raw data or refreshing the weather forecast).

To regenerate, run both notebooks top-to-bottom in order:

```
1. crowd_prediction_data_processing.ipynb
2. crowd_prediction.ipynb
```

This repopulates `data/processed/`, `data/master/`, and `models/`.

### Step 2 — Start the server

```bash
cd api
uvicorn main:app --reload --port 8000
```

The `--reload` flag restarts the server automatically when you edit any `.py` file.

### Step 3 — Verify startup

Check the terminal output. You should see all data loaded successfully:

```
Loading models...
  Lag baselines loaded for 90 (cell, period) pairs
  Training corridors: 30 unique cells
  Proxy scores computed for 524 cells (mean training proxy=19.7)
  Loaded tlc_h3_reference.csv (...)
  Loaded mta_h3_reference.csv (...)
  Loaded citibike_h3_reference.csv (...)
Ready — 81 features, 524 grid cells.
```

Then hit the health check:

```bash
curl http://localhost:8000/
```

Expected response:
```json
{
  "message": "Zentra Crowd Prediction API is running!",
  "docs": "/docs",
  "loaded": {
    "trained_cells": 30,
    "lag_baselines": 90,
    "proxy_cells": 524,
    "mean_train_proxy": 19.7,
    "feature_cols": 81
  }
}
```

If `proxy_cells` is 0, the live grid snapshot is missing — re-run notebook 1.  
If `trained_cells` is 0, the training master CSV is missing — re-run both notebooks.

## Expected Output Format

### `POST /predict/crowd` and `POST /predict/future`

```json
{
  "lat": 40.758,
  "lon": -73.9855,
  "h3_cell": "892a100d647ffff",
  "period": "EVE",
  "timestamp": "2026-07-04T20:00:00",
  "pedestrians": 9102.1,
  "crowd_score": 82.4,
  "crowd_category": "Very Busy",

  // future only:
  "days_ahead": 4.21,
  "weather_source": "forecast"
}
```

### `GET /predict/crowd-score`

```json
{
  "lat": 40.758,
  "lon": -73.9855,
  "h3_cell": "892a100d647ffff",
  "period": "EVE",
  "timestamp": "2026-07-04T20:00:00",
  "crowd_score": 82.4,
  "crowd_category": "Very Busy"
}
```

### Field Reference

| Field | Type | Description |
|-------|------|-------------|
| `lat` / `lon` | float | Echoed from request |
| `h3_cell` | string | Uber H3 resolution-9 hex cell ID (~150m grid) |
| `period` | string | Time bucket: `EARLY` / `AM` / `MD` / `PM` / `EVE` / `NIGHT` |
| `timestamp` | string | ISO 8601 datetime the prediction is for |
| `pedestrians` | float | Predicted pedestrian count in that corridor |
| `crowd_score` | float | 0–100 crowd intensity (0 = empty, 100 = maximum recorded) |
| `crowd_category` | string | Human-readable label based on crowd_score |
| `days_ahead` | float | Days from now to the predicted time (future endpoint only) |
| `weather_source` | string | `"forecast"` or `"seasonal_average"` (future endpoint only) |

### Crowd Category Thresholds

| crowd_score | crowd_category | Colour suggestion |
|-------------|----------------|-------------------|
| 0 – 19 | Quiet | Green |
| 20 – 39 | Light | Light green |
| 40 – 59 | Moderate | Yellow |
| 60 – 79 | Busy | Orange |
| 80 – 100 | Very Busy | Red |

### Period Time Ranges

| period | Hours (24h) |
|--------|-------------|
| `EARLY` | 00:00 – 06:59 |
| `AM` | 07:00 – 09:59 |
| `MD` | 10:00 – 13:59 |
| `PM` | 14:00 – 17:59 |
| `EVE` | 18:00 – 21:59 |
| `NIGHT` | 22:00 – 23:59 |

## Model Performance (Validation Set)

| Model | MAE | R² |
|-------|-----|----|
| LightGBM | 0.123 | 0.935 |
| HistGradientBoosting | 0.045 | 0.990 |
| RandomForest | 0.062 | 0.977 |
| **Ensemble (3-model blend)** | **~0.04** | **~0.93+** |

*Metrics are on log-transformed pedestrian counts. Training data: 30 Manhattan corridors, 2007–2025.*
